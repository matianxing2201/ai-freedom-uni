## Context

当前 `src/pages/me/me.vue` 仅是一个占位文本页面，而仓库已经具备 `token` / `user` store，可以提供登录态和用户信息状态。用户提供了一张已登录态设计稿，视觉上是浅绿渐变头部、圆角白色内容卡片、作品统计模块和服务入口列表；同时要求扩展一个与个人中心统一风格的未登录态，承接手机号登录、微信登录和注册流程，并参考 4.1-5.5 的接口文档。

## Goals / Non-Goals

**Goals:**
- 在同一个 `me.vue` 页面内实现已登录态与未登录态两套界面
- 已登录态尽可能按设计稿还原布局、色彩、圆角、阴影和图标使用
- 未登录态延续同一视觉语言，提供手机号验证码登录、微信登录、注册入口
- 将登录成功后的 token 与用户信息刷新接入现有 store
- 为后续账号设置、头像昵称修改、绑定手机号/邮箱预留 API 承接点

**Non-Goals:**
- 本次不实现完整的账号设置、帮助反馈、关于我们详情页，只负责入口导航
- 本次不实现复杂的资料编辑流程页面，只提供头像/昵称点击入口及接口承接设计
- 本次不引入新的 UI 组件库或第三方认证库

## Decisions

### 1. 页面结构采用“单页双态 + 局部弹层”

已登录态与未登录态共用 `src/pages/me/me.vue` 页面，根据 `tokenStore` / `userStore` 的状态计算 `isLoggedIn` 切换视图，而不是拆成两个独立页面。这样可以避免 tab 页切换时路由抖动，也更符合“我的”页作为 tab 页的使用方式。

### 2. 认证交互采用“页面主体 + 底部弹层表单”

未登录态展示品牌化欢迎区、手机号登录入口、微信登录按钮和“去注册”入口。点击手机号登录或注册时，打开底部弹层：
- 登录：手机号 + 验证码
- 注册：手机号 + 验证码 + 必要的确认信息

微信登录直接调用现有 `tokenStore.wxLogin()`。手机号流程需要新增专用 API 封装，与当前示例 `src/api/login.ts` 并存，不复用示例接口地址。

### 3. 作品统计卡使用本地图标和静态色块背景

设计稿中的“我的作品”区域包含 4 个业务卡片，使用 `src/static/images/mine` 下图标素材渲染，不依赖远程图标。卡片数字默认从用户任务数据或占位值渲染，若接口尚未提供统计聚合数据，则本次以本地占位 0/1 展示，接口字段到位后再替换。

### 4. API 分层采用单独 `src/api/me.ts`

为避免把项目示例 `src/api/login.ts` 与真实业务认证混在一起，新建 `src/api/me.ts`：
- `loginByMobileCode`
- `sendLoginCode`
- `queryUserInfo`
- `updateUserInfo`
- `bindMobile`
- `bindMail`
- `sendBindMsgCode`

如果当前后端微信登录参数和现有 `wxLogin` 示例不一致，则同步在该文件中补齐业务版微信登录接口。

### 5. 视觉风格采用“浅绿雾面渐变 + 大圆角白卡片”

已登录态严格遵循设计稿：
- 顶部区域为浅绿到薄荷色渐变
- 中部为白底大圆角卡片，轻阴影
- 作品统计卡使用低饱和色块背景与右下角业务图标
- 服务列表使用大留白与细分隔线

未登录态沿用同样的渐变头部和卡片体系，避免登录前后风格割裂。

## Risks / Trade-offs

- [接口字段与现有 store 类型不一致] → 通过 `src/api/me.ts` 单独定义业务接口类型，再在页面层做映射，避免污染现有示例类型
- [注册流程后端未明确单独接口] → 设计中将注册入口定义为手机号验证码注册流；若后端实际与登录共用接口，可在实现时合并为同一表单流程
- [作品统计缺少聚合接口] → 先按设计稿使用占位值渲染并保留可扩展字段，后续补真实统计来源
- [设计稿只提供已登录态截图] → 未登录态在同风格基础上自主设计，保持结构与品牌一致，但不要求逐像素还原不存在的稿件

## Migration Plan

1. 新建 change 对应实现所需 API 封装与页面组件
2. 重写 `src/pages/me/me.vue` 完成双态页面
3. 接入现有 token/user store 与新的业务认证 API
4. 在微信开发者工具中验证：未登录展示、验证码登录、微信登录、登录后切换到已登录态、退出后回到未登录态

回滚：还原 `src/pages/me/me.vue` 到占位版本，并移除新增 API 文件即可，无数据迁移。
